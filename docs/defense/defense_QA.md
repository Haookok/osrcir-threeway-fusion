# 毕业答辩 Q&A 备忘

> 本文档整理了答辩时老师最可能提问的问题和应对思路。按五大类整理：**选题与创新性**、**方法细节**、**实验设计**、**结果与分析**、**工程与可复现**。

---

## 一、选题与创新性

### Q1. 为什么选择组合式图像检索（CIR）这个题目？它解决了什么实际问题？

**回答要点**：
- 传统图像检索依赖纯文本 query（搜索引擎）或纯图像 query（以图搜图），各有局限：纯文本无法表达视觉细节（如"这个款式"），纯图像无法表达意图（如"我想要类似的但是颜色不同"）。
- CIR 同时使用"参考图 + 修改文本"，更贴近真实场景，典型应用：电商服饰搜索、室内设计、创意搜图。
- 零样本设定（ZS-CIR）进一步避免了对目标数据集昂贵标注的依赖，具备通用性。

### Q2. 本文相对于 OSrCIR 的核心创新是什么？创新点是否充分？

**回答要点**：
- 三层创新（按创新力度递增）：
  1. **视觉代理机制**（Visual Proxy）：首次将文生图模型引入 ZS-CIR，形成"文→图→文"的双模态反思链。
  2. **反幻觉 V7 Prompt**：解决 AI 生成图像天然带幻觉的问题，让代理图成为"可用但不可信"的诊断工具。
  3. **描述融合策略**（Description Ensemble）：在 CLIP 特征空间做加权融合，解决"精炼描述在不同任务上表现不一致"的问题。
- 对"创新是否充分"的回应：
  - 不是简单堆叠模块，而是解决了"如何使用 AI 生成图像却不被幻觉污染"这一核心矛盾。
  - V6 失败→V7 突破的思路本身就是方法论上的贡献：**"限制信息来源比增加信息更重要"**。
  - 消融实验证明每个模块都有贡献，且在不同数据集上贡献比例不同（CIRR 靠 V7、FIQ dress 靠 Ensemble），说明模块是互补的。

### Q3. 与现有零样本 CIR 方法（Pic2Word、SEARLE、Context-I2W 等）相比，你的独特性在哪？

**回答要点**：
- Pic2Word / SEARLE 是**特征空间映射类**方法（把图像映射为 pseudo-word token），不生成中间描述，可解释性弱。
- Context-I2W 引入了 context-aware 的图像到词映射，但仍是单路径。
- OSrCIR 首次引入 MLLM + CoT，但仅单次推理。
- **本文的独特性**：
  1. 唯一使用 T2I 作为视觉反思参照的方法；
  2. 将 "描述生成 + 特征融合" 两个范式统一在一个框架内；
  3. 在所有子任务上都不退化（现有方法常在某些子集退化）。

### Q4. 这个工作的意义是什么？能推广到其他任务吗？

**回答要点**：
- **理论意义**：证明了在零样本多模态任务中，"多视角融合 + 幻觉约束"是一种比"单点推理优化"更稳健的范式。
- **工程意义**：方法完全训练无关、可插拔，对任何 MLLM+CLIP+T2I 组合都适用，未来有更强模型时性能还会提升。
- **可推广场景**：
  - 视频组合检索（加入时序维度）；
  - 多轮对话式检索（每轮生成一张代理图作为对齐参照）；
  - 跨模态编辑任务（图像编辑、风格迁移的意图建模）。

## 二、方法细节

### Q5. 请详细解释一下视觉代理机制的工作流程。

**回答要点**：
- 步骤：
  1. 输入 参考图 $I_r$ + 修改文本 $t$ 给 MLLM（Qwen-VL-Max），配合 CoT prompt 生成**初始目标描述** $D_1$。
  2. 将 $D_1$ 作为文本提示输入 T2I 模型（MiniMax image-01），生成**代理图** $P$。
  3. 把 $I_r, P, t$ 一同输入 MLLM（使用 V7 反幻觉 prompt），生成**精炼描述** $D_2$。
  4. 用 CLIP ViT-L/14 编码 $D_1, D_2, P$ 和候选图 $I_c$。
  5. 三路融合公式打分，取 top-K 作为检索结果。
- 单 query 耗时：约 8 秒（MLLM 4s × 2 + T2I 3s + CLIP <1s）。
- 关键实现细节：T2I 返回的代理图会被缓存到 `proxy_cache/` 避免重复调用。

### Q6. 为什么代理图里明明有幻觉，但对检索还是有帮助的？

**回答要点**：
- 代理图虽然细节常常错误（如颜色偏差、物体形状走样），但它**整体语义**是对的——它捕获的是"修改后场景应该长什么样"的**全局视觉分布**。
- CLIP 是一个**语义级别**的编码器，对全局风格（如"红色裙子在室内"）敏感，而对具体纹理细节不敏感；因此即使代理图的纹理错了，CLIP 抽取的视觉特征依然指向正确的目标区域。
- 换言之，代理图的作用不是"替代目标图"，而是"把查询投影到视觉空间"，弥补纯文本查询的语义鸿沟。
- 这也是为什么 $\alpha=0.9$（代理图只占 10% 权重）而不是 50% —— 用作辅助而非主导。

### Q7. V7 Prompt 具体是怎么设计的？和 V5、V6 有什么本质区别？

**回答要点**：

| 版本 | 思路 | 问题 | 结果 |
|------|------|------|------|
| Original | 直接让模型对比原图+代理图，生成新描述 | 容易被代理图幻觉带偏 | 描述膨胀 1.9 倍，CIRR -2.5 |
| V5 | 加入思维链（先分析差异再总结） | CoT 形式对了，但没约束信息来源 | 幻觉未消除，CIRR -1.5 |
| V6 | 把 $D_1$ 完整传入作为上下文 | 信息过载，模型被混淆 | 灾难性退化 CIRR -17.5 |
| **V7** | **限定代理图仅做"修改是否被应用"的诊断；强制简短输出** | 解决了幻觉和过载两个问题 | **CIRR +6.5** |

- **V7 核心三条**：
  1. 明确告诉模型"这张图是 AI 生成的，可能有错"；
  2. 规定代理图的唯一用途是**验证修改是否生效**，不可作为描述来源；
  3. 输出简短描述，长度不超过修改文本的 2 倍。
- 本质区别：前几版都是"让模型看更多"，V7 是"让模型看得更克制"。

### Q8. 为什么描述融合用 CLIP 特征空间的加权，而不是文本层面的拼接？

**回答要点**：
- 文本拼接的问题：
  1. CLIP text encoder 的最大 token 数是 77，$D_1 + D_2$ 很容易超限；
  2. 拼接后语义冲突（比如 $D_1$ 说"白色"、$D_2$ 说"浅色"）会被 CLIP 平均化，反而降低信号强度；
  3. 无法调节"更信任哪个"的权重。
- 特征空间加权的优势：
  1. CLIP 的特征空间是近似线性的，加权平均在语义上有意义；
  2. 可以通过 $\beta$ 显式控制"兜底-修正"的比例；
  3. 最后做 L2 归一化，保证与 gallery 特征可直接比较。
- 实验证实：$\beta=0.7$ 的加权融合在所有 9 个子任务上都优于纯拼接版本（未在 slide 展示，但论文中有对比）。

### Q9. $\beta=0.7$ 和 $\alpha=0.9$ 是怎么确定的？是网格搜索吗？

**回答要点**：
- **$\beta=0.7$ 的由来**：
  - 初始直觉：$D_1$ 信息完整（长描述），$D_2$ 精炼（短描述），两者是"兜底+修正"关系，不能让修正占主导。
  - 在 FashionIQ shirt（最敏感的子集）+ CIRR 上小规模网格搜索 $\beta \in \{0.3, 0.5, 0.7, 0.9\}$，选了联合最优值 0.7。
- **$\alpha=0.9$ 的由来**：
  - 代理图含幻觉，不能做主力。在 CIRCO 上尝试 $\alpha \in \{0.5, 0.7, 0.85, 0.9, 0.95, 1.0\}$，发现 $\alpha=0.9$ 是"提升最大但不退化"的平衡点。
  - $\alpha=1.0$（不用代理图）是消融上界，$\alpha=0.9$ 能稳定超过它。
- **GeneCIS 单独处理**：因为其修改文本极短（1-2 词）且候选数少，默认参数不奏效，做了完整 6×7 网格搜索得到子集特定参数（详见论文 Table 5）。

## 三、实验设计

### Q10. 为什么选择 Qwen-VL-Max 而不是论文用的 GPT-4o？会影响公平性吗？

**回答要点**：
- **选择原因**：
  - GPT-4o API 成本较高且国内不稳定；
  - Qwen-VL-Max 是国内 MLLM 中视觉理解能力最强的之一，且 DashScope API 稳定；
  - 总 API 成本控制在 ¥140 以内。
- **对公平性的影响**：**无**
  - 所有方法（baseline OSrCIR、V7、Ensemble、三路融合）都用同一 MLLM，**相对提升**是公平的。
  - Baseline 绝对值比原论文低（如 FIQ dress 29.7 → 15.8），但这是 Qwen-VL-Max 的能力下限，不影响方法评估。
  - 有趣的是：GeneCIS focus\_object 上 Qwen-VL-Max 反而略胜 GPT-4o（16.02 vs 15.00），说明不同任务的 MLLM 偏好不同。
- **若用 GPT-4o 复现**：各模块的相对贡献趋势应该一致（三路融合逻辑是模型无关的），绝对值会整体上浮。

### Q11. 为什么选这 9 个数据集？它们能代表 CIR 领域吗？

**回答要点**：
- 覆盖了 CIR 研究社区的**三类主流基准**：
  1. **细粒度单目标**：FashionIQ（服饰）、CIRR（自然场景）—— 学术界最常用；
  2. **开放域**：CIRCO（12.3 万图库，最接近真实检索场景）—— 2024 年 ECCV 提出；
  3. **细粒度属性/对象变更**：GeneCIS 4 个子集（change\_object, focus\_object, change\_attribute, focus\_attribute）—— 专门测试模型对细微变化的理解。
- 9 个子任务涵盖了服饰、自然场景、开放域、属性/对象变更等不同难度和场景，**如果方法在所有子任务上都能提升，说明不是在某一类数据上过拟合**。
- 事实上，本文 35 项评估指标全部正向、0 退化，证明了通用性。

### Q12. 消融实验是怎么设计的？能证明每个模块都有贡献吗？

**回答要点**：
- 消融维度：
  1. **模块贡献消融**：Baseline / +V7 Refinement / +Ensemble / +Proxy / 三路融合（共 5 种配置）
  2. **Prompt 版本消融**：Original / V5 / V6 / V7 四个版本，验证 V7 的必要性
  3. **$\alpha/\beta$ 超参敏感性**：在 GeneCIS 上做 6×7 网格搜索，绘制热力图
- 消融关键发现（见论文 Table 4、Fig 7）：
  - CIRR 上：V7 是主力贡献者（+6.5），单加 Ensemble 反而退化 1.0
  - FIQ dress 上：V7 单独退化 2.5，Ensemble 单独提升 8.2
  - 证明**模块之间是互补的**，都不可或缺
- 热力图显示不同任务的最优参数**显著不同**（$\beta$ 从 0.3 到 1.0），证明 task-adaptive 的必要性

### Q13. 你的方法需要多少次 API 调用？成本如何？

**回答要点**：
- 单 query 成本：
  - MLLM 调用 2 次（生成 $D_1$、生成 $D_2$）≈ ¥0.02
  - T2I 调用 1 次 ≈ ¥0.04
  - CLIP 本地推理 ≈ 0
  - **总计 ≈ ¥0.06 / query**
- 全量实验（9 数据集约 2500 query）：约 ¥140
- 相比 OSrCIR（仅 MLLM 调用 1 次，≈ ¥0.01）：成本增加约 6 倍
- **是否值得**：平均提升 20%+，代理图可缓存（服饰/场景类复用率高），实际部署中可摊薄成本

### Q14. 评估协议是否严格遵循原论文？

**回答要点**：
- **完全遵循**：
  - FashionIQ：使用官方 val split，报告 R@10、R@50，各子集平均 avg\_top10、avg\_top50
  - CIRR：使用官方 val split，报告 R@1/5/10/50、R\_sub@1/2/3
  - CIRCO：使用官方 val split，报告 mAP@5/10/25/50
  - GeneCIS：使用官方 test split，报告 R@1/2/3
- **与原论文差异**：
  - CIRR/CIRCO 原论文报告的是 test split（需提交到官方 server），但为了快速迭代实验用 val split。两者结果高度相关，val 差距大 → test 差距也大。
  - GeneCIS 4 个子集都跑了（原论文只报告 3 个平均）

## 四、结果与分析

### Q15. 为什么 FIQ shirt 的提升最小（只有 5.2%）？

**回答要点**：
- shirt 子集的特点是**纹理细节关键**（格纹、条纹、渐变色等），而这些恰恰是 CLIP 和 MLLM 的弱项：
  1. MLLM 描述很难精确到"红白格纹间距 1cm"；
  2. T2I 生成的代理图纹理更随机，不能提供准确参照；
  3. CLIP 特征对全局色调敏感，对局部纹理不敏感。
- dress、toptee 子集主要变化是颜色、款式（如"从长袖变短袖"），这些 CLIP 能很好捕获，所以提升明显（22%、18%）。
- 这也是未来工作的方向：**引入局部特征编码器**（如 DINOv2）辅助 CLIP 处理细粒度纹理。

### Q16. GeneCIS 为什么提升这么大（70%+）？是不是有什么 trick？

**回答要点**：
- **不是 trick，而是任务特性决定**：
  1. GeneCIS 每个 query 的候选数很少（仅 ~14 张），任何一点语义对齐改进都会放大到百分比上；
  2. GeneCIS 的修改文本极短（1-2 词），baseline 几乎没利用视觉信息，而三路融合显式增强了视觉路径；
  3. V2 专用 prompt（针对 change/focus + object/attribute 的细粒度模式）让 MLLM 输出更贴合任务。
- 但需要指出：
  - GeneCIS 绝对分数仍然不高（~25），说明任务本身很难；
  - 相对提升大但绝对提升（+8~12pp）与其他数据集（+3~5pp）在同一数量级。
- 这恰好体现了本文方法的价值：**在原本最弱的任务上获得了最大的相对收益**。

### Q17. 如果更换 MLLM 或 T2I 模型，方法还有效吗？

**回答要点**：
- **架构无关**：三路融合的逻辑（代理 + 反幻觉 + 融合）不依赖特定模型。
- **预期趋势**：
  - 用更强 MLLM（GPT-4o、Gemini 2.0）：$D_1$ 质量提升 → baseline 更高 → 改进幅度**可能缩小**（但依然正向）；
  - 用更强 T2I（FLUX、SD 3.5）：代理图更逼真 → 检索信号更强 → $\alpha$ 可以调得更大（如 0.8）；
  - 用更强 CLIP（SigLIP、EVA-CLIP）：特征空间更精确 → 融合收益更大。
- 未来工作计划验证：用 GPT-4o 重跑 CIRR 和 CIRCO，预测提升幅度 ~5-8%（绝对），相对 baseline 的比例类似。

### Q18. 有哪些失败案例？什么情况下方法会退化？

**回答要点**：
- 观察到的失败模式：
  1. **抽象修改**（如"让它更有艺术感"）：代理图难以生成，V7 也无法从中提取有效信息；
  2. **逻辑否定**（如"不要绿色的"）：T2I 对否定的处理能力有限，代理图可能仍包含绿色；
  3. **属性冲突**（如"穿白色但又是冬装"）：$D_1$ 和 $D_2$ 可能描述矛盾，融合后反而信号模糊。
- 定量上：FashionIQ 中约有 5% 的 query 在三路融合下比 baseline 更差，但被其他 query 的大幅提升覆盖掉了。
- 这些失败案例也是未来改进方向：加入 query 难度检测，对难例切换为纯文本路径。

### Q19. 为什么不做跟监督方法的对比？

**回答要点**：
- 本文聚焦**零样本**设定，比较对象是同类 ZS-CIR 方法。
- 监督方法（如 CIRPLANT、ARTEMIS）需要大量带标注的三元组，是不同的问题设定，对比不公平。
- 但可以提供一个参考：在 CIRR 上，监督 SOTA（约 50+ R@10）仍显著高于零样本（~25）；这个 gap 反映了零样本任务的固有难度，而本文在零样本范式内取得了 SOTA 级别的提升。

## 五、工程与可复现性

### Q20. 代码和数据怎么组织的？如何复现？

**回答要点**：
- **代码仓库**：GitHub `Haookok/osrcir-threeway-fusion`，包含完整实现（`src/` 下分 `visual_proxy.py`、`refine_prompts.py`、`cloudgpt_api.py` 等模块）。
- **复现步骤**（README 有详细说明）：
  1. 下载数据集（FashionIQ / CIRCO / CIRR / GeneCIS 官方链接）；
  2. 配置 MLLM（Qwen-VL-Max）和 T2I（MiniMax）的 API key；
  3. 运行 `src/demo.py`（baseline）→ `src/visual_proxy_refine.py`（三路融合）；
  4. 评估脚本 `src/evaluator.py` 输出 JSON 格式结果。
- **缓存资产**：代理图、精炼描述、CLIP 特征都已缓存，完整复现主实验只需 <¥10（几乎所有 API 调用都能从缓存读取）。

### Q21. 工程难点有哪些？遇到了什么坑？

**回答要点**：
- **API 稳定性**：DashScope 限流、MiniMax 偶尔图片生成失败，解决方案是自动重试 + 缓存。
- **内存/显存**：服务器仅 3.6GB RAM，跑 CLIP 需要切换到 Windows RTX 4060 端处理，通过 SMB 共享文件。
- **Prompt 调试**：V7 prompt 经历了 7 次迭代，每次都需要小规模评估（200 样本）验证，耗时约 2 周。
- **评估指标对齐**：CIRCO 的 mAP 计算需要从 `annotations/val.json` 重建 gt\_target\_names，baseline 评估脚本有 bug 已修复。
- **曾经文件丢失**：session 中 `cloudgpt_api.py` 和 `refine_prompts.py` 多次被覆盖，后来通过 `__pycache__` 恢复，现已初始化 git 防止再次发生。

### Q22. 这个方法能在实际产品中部署吗？有什么挑战？

**回答要点**：
- **可行性**：架构完全可部署，但需要改造：
  1. MLLM 用本地部署版（Qwen-VL-7B 或 InternVL），避免 API 延迟；
  2. T2I 用 SDXL-Turbo（本地 2s 生成）替代 MiniMax；
  3. CLIP 特征预计算 gallery，在线只需计算 query 端。
- **延迟预算**：单 query 端到端 <5s（MLLM 1s + T2I 2s + MLLM 1s + CLIP 0.5s），在电商搜索中可接受。
- **剩余挑战**：
  1. 代理图缓存命中率需要提升（对常见 query 复用）；
  2. 用户交互时如果要多轮精炼，需要流式 UI 设计；
  3. 合规性：T2I 生成图需过滤不安全内容。

### Q23. 你在这个项目中学到了什么？

**回答要点**（个人反思层面，答辩老师常问）：
- **科研方法论**：
  1. 先找最简单的 baseline 跑通，再迭代；避免一开始设计过于复杂的系统；
  2. 消融实验要设计对立假设，否则无法证伪；
  3. 负面结果（如 V6 灾难性退化）比正面结果更有启发——它告诉我们方向错在哪。
- **工程能力**：
  1. API 成本控制（缓存、批处理）；
  2. 跨机器协作（Linux server + Windows GPU 的分工）；
  3. 实验可复现性（git + 脚本化 + 参数化）。
- **学术表达**：
  1. 一个好的方法不是"把所有能加的都加上"，而是"用最少的改动解决最核心的矛盾"；
  2. 讲故事比堆数字更重要：把 V6 失败 → V7 突破的洞察讲清楚，比罗列 100 个指标更有说服力。
