## 二、数据集处理

### 2.1 FashionIQ

- **原始 split**：captions/*.json + image_splits/*.json
- **图库构造**：每个子集（dress/shirt/toptee）独立 gallery
- **特殊处理**：captions 是一个 query 对应**两句话**（`candidate.png`"turn red then change length"），本工作把两句拼接成单条修改文本

### 2.2 CIRCO

- **split**：val.json 作为测试集（220 query）
- **gallery**：val 集全部图片（123403 张，含共享的 COCO 2017 unlabeled）
- **关键发现**：baseline JSON 中无 `gt_target_names` 字段，mAP 需从 `annotations/val.json` 提取 ground truth

### 2.3 CIRR

- **split**：cirr/captions/cap.rc2.val.json（4181 query）
- **gallery**：同 val split（2297 张）
- 没有分 subset，全量评估

### 2.4 GeneCIS

- 4 个子集：change_object、focus_object、change_attribute、focus_attribute
- 每个子集 ~500 query
- **特殊**：每个 query 有自己的**局部 gallery（~14 张）**而非全局图库
- **原始 JSON 问题**：change_attribute 包含 base64 图片数据，400MB+。已创建 `slim200` 版本（仅 200 query，去除 base64）用于快速验证

## 三、方法实现细节

### 3.1 代理图生成流程

```python
# 伪代码
def generate_proxy_image(desc_D1, api_key):
    # MiniMax image-01 API
    response = requests.post(
        'https://api.minimax.chat/v1/image_generation',
        json={
            'model': 'image-01',
            'prompt': desc_D1,
            'aspect_ratio': '1:1',
            'response_format': 'url'
        },
        headers={'Authorization': f'Bearer {api_key}'}
    )
    return response.json()['data']['image_urls'][0]
```

**缓存策略**：代理图生成后本地保存（`proxy_cache/<dataset>/<query_id>.png`），避免重复 API 调用。

### 3.2 MLLM 精炼调用

```python
# Qwen-VL-Max with V7 prompt
messages = [
    {"role": "system", "content": V7_SYSTEM_PROMPT},
    {"role": "user", "content": [
        {"type": "image", "image": ref_image_url},
        {"type": "image", "image": proxy_image_url},
        {"type": "text", "text": f"修改文本: {modification}"}
    ]}
]
response = client.chat.completions.create(
    model="qwen-vl-max",
    messages=messages,
    temperature=0.0,      # 确定性
    max_tokens=200,
    seed=42
)
```

### 3.3 三路融合评分

```python
# 核心评分代码（伪代码）
def score(query_id, gallery_feats):
    # 两个描述的 CLIP 特征
    f_D1 = clip.encode_text(D1[query_id])  # (768,)
    f_D2 = clip.encode_text(D2[query_id])  # (768,)
    # 加权平均并归一化
    f_text = beta * f_D1 + (1 - beta) * f_D2
    f_text = f_text / f_text.norm()
    # 代理图的 CLIP 特征
    f_proxy = clip.encode_image(proxy_image[query_id])  # (768,)
    # 三路评分
    s_text = f_text @ gallery_feats.T    # (N,)
    s_img = f_proxy @ gallery_feats.T    # (N,)
    return alpha * s_text + (1 - alpha) * s_img
```

## 四、Prompt 演进详细记录

### 4.1 V0（Baseline，OSrCIR 原始）

- 单阶段反思式 CoT
- 仅使用参考图 + 修改文本
- 输出目标描述 D₁

### 4.2 V3（首次尝试加代理图）

- 在 V0 基础上加入代理图
- 没有明确使用规则
- 结果：CIRR R@10 从 67.0 → 62.5（-4.5）

### 4.3 V6（开放式代理图利用）

- Prompt: "Please refine the target description by referring to the proxy image"
- MLLM 把代理图当成"理想目标"，把其中所有细节抄进描述
- 结果：CIRR R@10 从 67.0 → 49.5（**-17.5，灾难性退化**）

### 4.4 V7（反幻觉约束，最终版）

关键 prompt 片段：
```
【重要警告】Image 2 是 AI 生成的代理图像，可能包含幻觉。
- ✓ 你可以用它来检查修改是否被正确应用（颜色、物体、属性）
- ✗ 禁止复制 Image 2 中的细节（背景、材质、光照、环境）
- ✗ 禁止描述 Image 2 中存在但参考图和修改文本中没有提到的内容

【输出要求】
- 简短目标描述（不超过修改文本长度的 1.5 倍）
- 专注于与检索相关的核心属性
```

结果：CIRR R@10 = 73.5（**+6.5**）

### 4.5 V2-GeneCIS（GeneCIS 专用）

GeneCIS 的 query 文本过短，V7 会导致描述太短。V2 强制要求 5-12 词 + 具体视觉特征：
```
For GeneCIS: generate a 5-12 word description with concrete visual attributes
(material, color, specific object parts).
```

## 五、调试和踩坑记录

### 5.1 OOM（服务器内存不足）

- **现象**：`torch.cuda.OutOfMemoryError` + 进程被 OOM killer 杀死
- **原因**：3.6GB RAM 不足以加载 CLIP ViT-L/14
- **解决**：把 CLIP 编码移到 Windows GPU，服务器仅做 API 调用和文本处理

### 5.2 openai 版本兼容

- **现象**：服务器 openai 2.29.0 语法和 0.28 不一致
- **解决**：`cloudgpt_api.py` 加兼容层，同时支持 `openai.ChatCompletion.create` 和 `client.chat.completions.create`

### 5.3 CIRCO 评估缺 ground truth

- **现象**：baseline JSON 里无 `gt_target_names` 字段，无法计算 mAP
- **解决**：从 `annotations/val.json` 提取每个 query 的 ground truth IDs

### 5.4 SSH session vs 桌面会话

- **现象**：Windows SSH 登录后看不到 SSHFS/SMB 挂载的 Z:/Y: 盘
- **原因**：SSHFS/SMB 是桌面会话的用户级挂载，SSH session 是另一个会话
- **解决**：用 `schtasks` 创建定时任务在桌面会话中执行

### 5.5 文件丢失

- **cloudgpt_api.py** 和 **refine_prompts.py** 在多次 session 切换中丢失过
- **解决**：
  - 从 `__pycache__/*.pyc` 恢复（uncompyle6）
  - 养成习惯每改动后立即 git commit（后来初始化了 git 仓库）

### 5.6 代理图生成超时

- **现象**：MiniMax API 偶尔超时（>30s）
- **解决**：重试机制（最多 3 次，每次间隔 5 秒），失败的 query 标记为 skip，退化为纯 D₁

## 八、可能被质疑的点及应对

| 质疑 | 应对 |
|---|---|
| "为什么不用 GPT-4o" | 访问受限，Qwen-VL-Max 是公平替代 |
| "代理图有幻觉怎么能用" | V7 + α=0.9 双重约束 |
| "35/35 是不是过拟合" | 所有对比都在固定数据上，无训练 |
| "β=0.7 看起来是拍脑袋" | 基于方法论假设 + 小样本验证不敏感 |
| "这个方法部署可行吗" | 离线建索引场景可行；在线需模型本地化 |
| "与其他 2026 方法比呢" | OSrCIR 2025 CVPR Highlight 已是前沿 |
| "做了多少消融" | Prompt 演进 + Ensemble 对比 + α/β 搜索三组 |
| "35 个指标怎么挑的" | 每数据集官方主指标，不挑数据 |
| "为什么 shirt 提升最小" | 修改文本最短，输入信息量有限（详见 C3）|
